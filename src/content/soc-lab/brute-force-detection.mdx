## Overview

In this writeup, I document how I set up and analyzed a simulated SSH brute force attack in my home lab using Splunk as the SIEM. The goal was to understand the log patterns generated by brute force attempts and build detection rules.

## Lab Environment

- **SIEM:** Splunk Free (local install)
- **Target:** Ubuntu 22.04 VM
- **Attacker:** Kali Linux VM on isolated network segment
- **Log source:** `/var/log/auth.log` forwarded via Splunk Universal Forwarder

## The Attack

I used `hydra` on the Kali machine to simulate credential stuffing against the Ubuntu host's SSH service. The command:

```bash
hydra -L /usr/share/wordlists/usernames.txt -P /usr/share/wordlists/rockyou.txt ssh://192.168.1.50
```

## What the Logs Showed

Each failed attempt generated an `auth.log` entry like:

```
May 1 14:22:03 ubuntu-lab sshd[12345]: Failed password for invalid user admin from 192.168.1.200 port 54321 ssh2
```

In Splunk, I searched for these with:

```spl
index=main sourcetype=syslog "Failed password"
| stats count by src_ip, user
| sort -count
```

## Detection Rule

I built an alert that fires when any single source IP generates more than 10 failed SSH attempts within 60 seconds — a reasonable threshold that catches brute force without alerting on fat-fingered passwords.

## Key Takeaways

- Brute force attacks are *loud* — they generate distinctive patterns that are easy to spot once you know what to look for.
- Rate limiting at the firewall level (`iptables` or `fail2ban`) should be the first line of defense, with SIEM alerts as the backstop.
- Setting up realistic lab scenarios is the fastest way to build pattern recognition.
